## Iterative + Timeboxing

### Development Approach: Iterative Delivery

**Philosophy:** Ship working increments every 1–3 days. Each cycle has a tangible deliverable.

**Rules:**

1. Phases in `development-plan.md` are 1–3 day cycles, not milestones. Each phase name is a deliverable, not a category.
2. When planning features (`/plan`), define a "done in N days" timebox for each task. Tasks that don't fit in 3 days must be split into smaller tasks.
3. Execution reports must include cycle duration and whether the timebox was met.
4. Prioritize shipping a working increment over perfection. A deployed feature with rough edges beats a polished feature in a branch.
5. At the end of each cycle, the feature must be in a usable (even if incomplete) state. No cycle ends with broken or non-functional code.

**Phase structure in `development-plan.md`:**

```
| # | Phase (Deliverable)              | Status  | Timebox |
|---|----------------------------------|---------|---------|
| 1 | Auth flow (login + signup)       | Pending | 2 days  |
| 2 | Dashboard with data grid         | Pending | 3 days  |
| 3 | User settings page               | Pending | 1 day   |
```

**How to split tasks:** If a task feels like it needs more than 3 days, ask: "What is the smallest slice that delivers value on its own?" Extract that slice as the current cycle and push the rest to subsequent cycles.

## Shape Up

### Development Approach: Shape Up

**Philosophy:** 6-week build cycles, 2-week cooldown. Work on appetites (how much time we're willing to spend), not estimates.

**Rules:**

1. Phases are "bets" with an appetite (1w, 2w, or 6w). There is no infinite backlog — work that isn't bet on is discarded.
2. When planning features (`/plan`), ask for appetite upfront: "How much time are we willing to spend on this?" The answer defines a hard ceiling, not a target.
3. Prefer scope reduction over incomplete features. When work exceeds appetite, suggest scope cuts instead of timeline extensions. Never extend the timeline.
4. Every bet must have a clear "hill chart" progression: unknown territory (uphill) → known territory (downhill). Tasks should be ordered so unknowns are resolved first.
5. Cooldown periods (2 weeks between cycles) are for bug fixes, refactoring, and exploration — not feature work. Do not schedule feature bets during cooldown.

**Phase structure in `development-plan.md`:**

```
| # | Bet                              | Status      | Appetite  |
|---|----------------------------------|-------------|-----------|
| 1 | Real-time notifications          | In Progress | 2 weeks   |
| 2 | Advanced search                  | Pending     | 1 week    |
| 3 | Cooldown: bug fixes & refactors  | Pending     | 2 weeks   |
```

**Scope hammering:** When a bet is running over appetite, do not ask for more time. Instead, list what can be cut or deferred to make the bet fit. The question is always "What can we drop?" — never "How much longer do we need?"

## TDD-First

### Development Approach: Test-Driven Development

**Philosophy:** Tests are written before implementation. Test coverage is the primary quality signal.

**Rules:**

1. Every feature task follows red-green-refactor:
   - **Red:** Write a failing test that describes the expected behavior.
   - **Green:** Write the minimum implementation to make the test pass.
   - **Refactor:** Clean up the code while keeping all tests green.
2. Plans must include a "Test Strategy" section defining test types (unit, integration, e2e) and coverage targets per task.
3. Write tests before implementation — this is directive #1 for all work. No exceptions.
4. Never skip tests to save time. If a feature cannot be tested, reconsider the design until it can be.
5. Test coverage must not decrease after any task completion. If new code is added, new tests must accompany it.
6. Tests are documentation. Each test name should read as a specification: `should reject login with invalid credentials`, not `test login`.

**Task execution order:**

```
1. Read the task requirements
2. Write test cases that define "done"
3. Run tests — confirm they fail (red)
4. Implement the feature
5. Run tests — confirm they pass (green)
6. Refactor implementation and tests
7. Run tests — confirm they still pass
8. Check coverage has not decreased
```

**Test naming convention:** Use descriptive names that express intent. A reader should understand what the feature does by reading test names alone.

## Trunk-Based

### Development Approach: Trunk-Based Development

**Philosophy:** Single main branch, short-lived feature branches (max 1 day), feature flags for work in progress.

**Rules:**

1. Tasks must be mergeable independently. No task should require another unmerged task to function.
2. Feature branches must not live longer than 1 day. Branches older than 1 day should be flagged for immediate attention and either merged or rebased.
3. Large features use feature flags rather than long-lived branches. The flag allows code to be merged to main while the feature remains hidden from users.
4. Commit after each logical change, not at the end of a session. Small, frequent commits reduce merge conflicts and make history useful.
5. After each completed task: commit, push, and verify that main remains deployable.
6. Every commit must leave main in a deployable state. If a commit breaks the build or tests, fixing it is the top priority before any other work.

**Branching workflow:**

```
1. Pull latest main
2. Create short-lived branch: feat/task-name
3. Make changes in small commits
4. Push and open PR (or merge directly for trivial changes)
5. Merge same day — do not let the branch age
6. Delete the branch after merge
```

**Feature flags:** For multi-day features, merge code behind a flag. The flag defaults to off in production. Remove the flag once the feature is fully shipped and stable.

## YAGNI/KISS

### Development Approach: YAGNI (You Aren't Gonna Need It)

**Philosophy:** Build the minimum that works. Refactor only when a second similar case appears.

**Rules:**

1. Every task must justify why it cannot be simpler. Apply a "Do we really need this?" gate before starting any work.
2. Prefer the simplest solution. Do not create abstractions, interfaces, or indirection for single use cases.
3. Three similar lines of code is better than a premature abstraction. Extract a shared function only when the third instance appears.
4. Flag tasks that look like premature abstraction or over-engineering. Examples: building a plugin system for one plugin, creating a config framework for three settings, adding an event bus for two events.
5. Flag files over 300 lines or functions over 50 lines as potential complexity issues that need review.
6. Do not design for hypothetical future requirements. Build for what is needed today. If a future requirement materializes, refactor then — not before.

**Decision checklist before adding complexity:**

```
- Is this solving a problem we have right now? (not "might have")
- Is there a simpler way to achieve the same result?
- Would a new developer understand this without explanation?
- Are we adding this because we need it or because it feels "correct"?
```

**When to refactor:** Refactor when you encounter the second real instance of a pattern — not the first, and not a hypothetical one. The rule of three applies: duplicate once is acceptable, duplicate twice means extract.
